给产品 / 工程 / 设计的直观版 LLM 原理图

LLM 的
技术原理

把它想成一台“会补全文的概率机器”。它先把文字切成 token，再把 token 映射成向量，用多层 Transformer 反复计算“当前最该关注谁”，最后逐 token 预测下一个最可能出现的符号。

整条链路
文本 → tokenization → embedding → attention → next token

这些柱状条只是在示意：词语会被映射成很多维的数值特征，而不是直接“理解”成概念卡片。

01
本质

LLM 并不是查字典回答，而是在上下文条件下做概率预测。

02
关键创新

Transformer 让模型能在一整段上下文里动态决定注意力权重。

03
为什么强

参数量、数据量、算力和训练技巧一起把“补全”放大成了推理能力。

文本离散 向量表示 上下文关联 概率采样
像白板课件，不像产品官网
Step 1
输入先被切块

Tokenization: 先把文字切成模型能处理的颗粒

不是“按词切”，而是按子词 / 字符片段切

大型 语言 模型 可以 生成 文本
trans form ers are powerful

这样做的原因是词表更可控，能覆盖新词、长尾词、拼写变化和多语言混合场景。

文本 ≠ 原子对象，token 才是模型看到的最小离散单位
为什么 token 数重要？
  • 决定上下文窗口消耗
  • 直接影响推理成本和延迟
  • prompt 工程本质上也在管理 token 预算
一个直觉

模型不是在“读句子”，而是在读一串编号序列，例如 [1532, 887, 42...]。

输入上限

当上下文超过窗口，模型就无法同时看见全部历史，于是需要截断、检索或压缩。

第一步只是“把语言编码成索引”，真正的语义处理还没开始。

切块不是理解
Step 2
token 变成向量

Embedding: 把离散 token 投影到连续空间

向量不是标签，而是一组坐标

每个 token 会对应到一个高维向量。语义相近、语法角色相近、上下文用法相近的 token，在空间里通常也更接近。

训练过程中，这些数值会持续被更新，最终形成可计算的“几何结构”。

位置也很重要

同样的 token 放在不同位置会有不同意义，所以要加入 positional encoding，让模型知道先后顺序。

直观理解

Embedding 像给每个 token 画了一张“特征画像”，接下来 attention 才会拿这些画像做比对。

从 here 到 meaning

词形特征 语法角色 上下文相关性 语义邻近性

这一步完成后，文本就进入了“神经网络可计算”的空间。

离散符号 → 连续向量
没有向量，就没有可微学习

相比 RNN 顺序传递状态，attention 能更直接地连接任意两个位置，更适合长上下文并行训练。

堆很多层会怎样？

浅层更偏词法和局部模式，深层更偏抽象语义、世界知识、任务格式和推理线路。

参数是什么？

就是这些矩阵里的可学习数值。所谓“175B 参数”，本质上是超大规模的可调系数集合。

一条粗暴但有效的经验

更大的模型 + 更大的数据 + 更长的训练，往往会冒出新的能力边界。

层数越多，表示越抽象
“会推理”常常是堆叠后的涌现结果
Step 5
模型是怎么学会的

Training: 先预训练，再对齐

1
海量语料

网页、书籍、代码、问答、文档等，组成大规模文本分布。

2
自监督目标

核心任务通常很朴素：根据前文预测下一个 token。

3
反向传播

预测错了就调整参数，让相似上下文下的概率分布更合理。

4
SFT / 指令微调

让模型更像“助理”，学会按任务格式回应，而不只是续写语料。

5
RLHF / DPO

用偏好数据把回答往“有用、无害、符合人类期待”方向推。

预训练学到什么

语言结构 模式压缩 部分世界知识 代码统计规律

对齐学到什么

遵循指令 拒绝危险请求 更自然的表达 符合产品体验
预训练决定“知道多少”
对齐决定“怎么说出来”
Step 6
线上生成发生了什么

Inference: 一次只生成一个 token，但循环得非常快

推理流程

prompt → logits → softmax → sampling → next token
  • Logits：模型对每个候选 token 的原始打分。
  • Softmax：把打分转成概率分布。
  • Sampling：按 greedy、temperature、top-k、top-p 等策略选下一个 token。
  • KV cache：缓存历史键值，避免每次从头重复算。
为什么 temperature 会影响风格？

温度高，概率分布更“平”，模型更敢选次优 token；温度低，输出更稳定保守。

为什么会流式输出？

因为模型本来就是按 token 逐步生成，UI 只是把这个过程实时暴露给用户。

速度瓶颈

推理的核心成本不只是参数量，还有显存带宽、上下文长度、并发和采样策略。

用户看到的是一句话，系统内部其实是几十到上千次“下一步选什么 token”的重复循环。

生成像滚动写字
Step 7
为什么好用，也为什么会错

LLM 的能力边界：强在模式归纳，弱在真实校验

它擅长的

  • 跨任务迁移：写作、摘要、分类、翻译、代码生成
  • 在上下文中学习格式和范式
  • 把复杂问题压缩成自然语言交互
  • 通过工具调用、RAG、工作流编排继续扩展能力

它容易出错的地方

  • 幻觉：概率上顺，但事实不一定真
  • 长链条推理脆弱，尤其在缺少外部校验时
  • 知识过时，除非有在线检索或工具接入
  • 对 prompt、上下文组织和采样设置很敏感
一句话总结

LLM 更像“会根据大量语料模式生成答案的系统”，而不是一个天然可靠的知识库。

工程上的正确姿势

把模型放进系统里，用检索、工具、规则、评测、反馈闭环约束它，而不是把它当作孤立万能脑。

产品判断标准

能否校验 能否回溯 能否约束 能否持续评测

真正可用的 AI 产品，不是“模型很聪明”，而是“系统能把模型的聪明变成稳定输出”。

模型负责生成，系统负责可信
